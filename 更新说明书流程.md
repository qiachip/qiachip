# 更新说明书流程

本文档描述了如何按照标准格式更新产品说明书的流程，以确保所有产品文档保持一致性和专业性。

**重要提醒**：
- 不能更改文件夹中的文档内容
- 只有在文档标题后面有 `(1)` 时，将其删除即可
- 例如：`# QIACHIP KR2201-4 Instruction Manual AC 110V 220V 433MHz RF Remote Control Switch 1-CH Relay Receiver (1)` 删除 `(1)` 后为：`# QIACHIP KR2201-4 Instruction Manual AC 110V 220V 433MHz RF Remote Control Switch 1-CH Relay Receiver`

## 1. 文件命名规范

### 标准命名格式
产品说明书的 Markdown 文件应遵循以下命名规范：
```
QIACHIP_[产品型号].md
```

### 示例
- 正确：`QIACHIP_KR1201A.md`
- 错误：`QIACHIP KR2202 Instruction Manual AC 110V 220V 433 2ac821bc22e7805eb1f8c51abe323303.md`

## 2. 文档格式规范

### 2.1 文件头部
每个 Markdown 文件应包含以下 YAML 前置元数据：
```yaml
---
comments: true
---
```

### 2.2 标题格式规范

**标题层级**：
- 一级标题（`#`）：仅用于文档主标题，如 `# QIACHIP KR2202 Instruction Manual`
- 二级标题（`##`）：用于主要章节，如 `## Product Size`、`## Component Description`、`## Wiring Diagram`、`## Electrical characteristics`、`## Warning` 等
- 三级标题（`###`）：用于子章节，如 `### Receiver`、`### Remote control`、`### Figure 1` 等
- 四级标题（`####`）：用于更细分的子章节

**重要说明**：
- `## Electrical characteristics` 和 `## Warning` 都是二级标题，不要把 `Warning` 变成三级标题
- 如果文章最后是 `NOTE` 而不是 `Warning`，那么 `## NOTE` 也应该是二级标题，与 `## Electrical characteristics` 同级
- 确保标题层级连续，不要跳级（如从二级直接跳到四级）
- 当调整一个标题的层级时，必须同时调整其下的所有子标题
- 移除标题中的粗体标记（如 `**标题**` 改为 `标题`）
- 确保同一级别的标题使用一致的格式

**参考文档**：`D:\WorkFiles\qiachipweb\docs\KR2202\QIACHIP_KR2202.md`

### 2.3 步骤标题格式
在功能描述和设置方法部分，步骤标题应使用加粗格式而非标题格式：

- 正确：`**Step 1**`、`**Step 2**`（加粗文本）
- 错误：`#### Step 1`、`#### Step 2`（四级标题）

**使用场景**：
- 功能描述中的模式设置步骤（Momentary mode、Toggle mode、Latching mode 等）
- 各种操作步骤说明
- 重置功能步骤

**格式要求**：
- 每个 **Step x** 标题后必须添加空行，使格式更清晰
- 确保步骤标题与后续内容之间有明确的分隔

**示例**：
```markdown
## Function description and setting method

### Momentary mode

**Step 1**

Press and hold the learning button on the receiver for 3 seconds.

**Step 2**

Press any button on the remote control.

**Step 3**

The LED indicator will flash 3 times to confirm pairing.
```

### 2.4 图片引用格式
所有图片引用应包含以下属性：
```markdown
![图片描述](图片路径){ width="50%" .center loading="lazy" }
```

- 产品图片使用 `width="50%"`
- 尺寸图使用 `width="68%"`
- 所有图片应居中显示并添加懒加载属性

### 2.5 尺寸数据格式
尺寸数据应使用小写字母 x 作为乘号：

- 表格中的尺寸数据：`115x53x25mm`（正确），避免使用 `115*53*25mm`（错误）
- 产品描述中的尺寸：`Length (L) x Width (W) x Height (H): 115mm x 53mm x 25mm`（正确），避免使用 `Length (L) × Width (W) × Height (H): 115mm × 53mm × 25mm`（错误）

### 2.6 组件描述格式
组件描述应使用 HTML 布局，分为两列显示：
```html
<div style="display: flex; flex-wrap: wrap; justify-content: space-around;">
  <ul style="flex: 1 1 45%; margin-right: 1%;">
    <li>组件1: 描述</li>
    <li>组件2: 描述</li>
  </ul>
  <ul style="flex: 1 1 45%; margin-left: 1%;">
    <li>组件3: 描述</li>
    <li>组件4: 描述</li>
  </ul>
</div>
```

### 2.7 表格标题
表格标题使用小写：
- 正确：`## Electrical characteristics`
- 错误：`# Electrical Characteristics`

## 3. 更新流程

### 步骤 1：准备工作
1. 确定需要更新的产品文档
2. 查看参考文档（`docs/KR2202/QIACHIP_KR2202.md`）了解标准格式
3. 检查当前文件名是否符合命名规范
4. 检查当前文件格式是否符合标准

### 步骤 2：重命名文件（如需要）
如果文件名不符合命名规范：
1. 使用标准命名格式重命名文件：`QIACHIP_[产品型号].md`
2. 确保文件名只包含产品型号，不包含额外描述
3. 注意：不要更改图片文件名，只更改 Markdown 文件名

### 步骤 3：更新文档格式

#### 3.1 标题层级调整（关键步骤）

**调整方法**：
1. 使用编辑器的大纲视图或搜索功能（搜索 `^#+\s` 正则表达式）识别所有标题
2. 按照标题层级规范调整每个标题的级别
3. 确保标题层级连续，不要跳级
4. **关键**：当调整一个标题的层级时，必须同时调整其下的所有子标题

**检查清单**：
- [ ] 主标题是否使用一级标题（#）
- [ ] 所有章节是否使用二级标题（##）
- [ ] 所有子章节是否使用三级标题（###）
- [ ] 是否存在标题层级跳级的情况
- [ ] 标题中是否还有粗体标记（**）
- [ ] **关键检查**：调整标题层级时，是否同步调整了所有子标题
- [ ] **关键检查**：是否使用大纲视图验证了整个标题结构

**验证技巧**：
- 使用 Markdown 编辑器的大纲视图功能查看整个文档结构
- 使用 VSCode 的 "Outline" 视图检查标题层级
- 与标准文档（KR2202）进行逐层对比
- 使用正则表达式搜索特定层级的标题：
  - 搜索 `^##\s` 查找所有二级标题
  - 搜索 `^###\s` 查找所有三级标题

#### 3.2 其他格式更新
1. 更新图片引用，添加所需属性：
   - 产品图片：`{ width="50%" .center loading="lazy" }`
   - 尺寸图：`{ width="68%" .center loading="lazy" }`
2. 调整组件描述为两列 HTML 布局
3. 检查并修正表格标题大小写（使用小写）
4. 将步骤标题从 `#### Step x` 改为 `**Step x**`
5. **在每个 **Step x** 标题后添加空行，使格式更清晰**

### 步骤 4：更新导航配置
1. 检查 `mkdocs.yml` 文件中的导航配置
2. 在 `nav` 部分添加新产品的导航条目
3. 格式：`- [产品型号]: [目录路径]/[文件名].md`
4. 确保导航条目按产品分类组织

### 步骤 5：更新首页链接
如果产品文档的文件名或图片路径发生变化：
1. 检查 `docs/index.md` 中的产品链接
2. 确保链接指向正确的新文件名
3. 确保图片路径与实际文件名一致
4. 验证所有产品分类下的链接都正确

### 步骤 6：验证更新
1. 运行 `mkdocs build --strict` 检查配置是否正确
2. 确保没有错误和警告信息
3. 检查文件格式是否与标准文档一致
4. 确保所有图片引用正确
5. 使用大纲视图验证标题层级结构
6. 启动开发服务器预览更改效果

## 4. 常见错误与解决方案

### 4.1 标题层级错误

| 错误类型 | 问题描述 | 解决方案 |
|---------|---------|---------|
| 标题层级不连续 | 从二级标题直接跳到四级标题 | 确保标题层级连续，按照 # → ## → ### → #### 的顺序使用 |
| 子标题未随主标题调整 | 将一级标题改为二级标题后，子标题未相应调整 | 调整标题层级时，必须同时调整其下的所有子标题 |
| 标题中保留粗体标记 | 标题中仍保留粗体标记，如 `## **标题**` | 移除标题中的粗体标记，改为 `## 标题` |
| 同级标题格式不一致 | 同一级别的标题使用了不同的格式 | 确保同一级别的标题使用一致的格式 |
| Warning 标题层级错误 | 将 `## Warning` 改为 `### Warning` | `## Electrical characteristics` 和 `## Warning` 都是二级标题 |

### 4.2 其他常见错误

| 错误类型 | 问题描述 | 解决方案 |
|---------|---------|---------|
| 图片引用缺少属性 | 图片引用未添加 width、center 和 loading 属性 | 按照规范添加所需属性 |
| 组件描述未使用两列布局 | 组件描述仍使用简单列表格式 | 使用 HTML 两列布局格式 |
| 步骤标题使用标题格式 | 步骤标题使用 `#### Step x` 格式 | 改为加粗格式 `**Step x**` |
| 步骤标题后缺少空行 | **Step x** 标题后没有空行，导致内容混淆 | 在每个 **Step x** 标题后添加空行，使格式更清晰 |

## 5. 注意事项

1. **保持内容不变**：更新过程中不应改变文档的实际内容，只调整格式
2. **图片路径**：确保图片引用路径正确，不要更改图片文件名
3. **一致性**：确保所有产品文档使用相同的格式标准
4. **备份**：在更新前建议备份原始文件，以防意外
5. **标题层级特别注意**：
   - 调整标题层级时，必须同时调整其下的所有子标题
   - 使用标题层级检查清单逐项检查
   - 对比参考文档（KR2202）的标题结构
   - 特别注意 `## Electrical characteristics` 和 `## Warning` 都是二级标题

### 6.1 标题层级示例

**更新前（错误）**：
```markdown
# QIACHIP FLC05-E220V Instruction Manual

## Product Size

# Component Description  <!-- 错误：应为二级标题 -->
## Receiver  <!-- 错误：应为三级标题 -->
## Remote control  <!-- 错误：应为三级标题 -->

# Wiring Diagram  <!-- 错误：应为二级标题 -->
## Figure 1  <!-- 错误：应为三级标题 -->

# Function description and setting method  <!-- 错误：应为二级标题 -->
## **(1) Pairing**  <!-- 错误：应为三级标题且不应有粗体 -->
## **How to Pair**  <!-- 错误：应为三级标题且不应有粗体 -->
```

**更新后（正确）**：
```markdown
# QIACHIP FLC05-E220V Instruction Manual

## Product Size

## Component Description
### Receiver
### Remote control

## Wiring Diagram
### Figure 1

## Function description and setting method
### (1) Pairing
### How to Pair
```

## 7. 最佳实践

### 7.1 使用工具辅助
- 利用编辑器的大纲视图功能
- 使用正则表达式搜索特定层级标题
- 与标准文档进行结构对比

### 7.2 遵循系统性流程
- 修改前先分析整个文档结构
- 修改时同步调整所有相关标题
- 修改后进行全面验证

### 7.3 使用详细检查清单
- 确保每个检查项都完成
- 特别关注"子标题同步调整"检查项
- 定期更新检查清单以适应新需求

### 7.4 建立质量保证机制
- 将标题层级检查作为必要步骤
- 定期审查和更新流程文档
- 分享经验教训，持续改进

通过遵循这些最佳实践，我们可以确保文档标题层级的一致性，提高文档质量，减少错误和返工，最终提升用户体验和团队效率。
